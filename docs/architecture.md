# Архитектура Проекта

Система спроектирована как web-приложение с разделением пользовательского интерфейса, серверного API, хранилища данных и асинхронного расчетного контура.

Архитектура построена вокруг простой идеи: пользователь работает только с удобным web-интерфейсом, сервер принимает и проверяет команды, база данных хранит факты предметной области, а длительные расчеты выполняются отдельно от обычных HTTP-запросов. Благодаря этому интерфейс остается отзывчивым, расчетный модуль можно развивать независимо, а данные не смешиваются с кодом отображения.

## Общая Схема

```text
Пользователь
    |
    v
React UI
    |
    v
C++ Drogon REST API
    |
    +--------------------+
    |                    |
    v                    v
PostgreSQL         Redpanda/Kafka
                         |
                         v
                    C++ Worker
                         |
                         v
                    PostgreSQL
```

Если описывать схему словами, то пользователь открывает web-приложение в браузере и выполняет действия через разделы "Карта", "Данные", "Расчеты", "Детали", "Отчеты", "Справочники" и "Статистика". Frontend отправляет запросы в REST API. API не выполняет сложные вычисления прямо в запросе: для обычных операций он сразу работает с PostgreSQL, а для расчета загрязнения создает запись задания и публикует событие в очередь. Worker получает это событие, выполняет расчет поля концентрации и сохраняет результат обратно в PostgreSQL. После этого frontend может запросить статус и рассчитанные точки.

Такая схема близка к практическому backend-подходу: синхронные действия выполняются через HTTP, а тяжелая работа передается в асинхронный контур.

## Архитектурный Стиль

В проекте используется слоистая архитектура с отдельным асинхронным обработчиком заданий.

Основные принципы:

- пользовательский интерфейс не знает SQL и не подключается к базе напрямую;
- REST API является единой точкой входа для клиента;
- бизнес-логика расчета вынесена в доменный слой;
- работа с PostgreSQL и Kafka находится в инфраструктурном слое;
- worker выполняет долгие операции отдельно от HTTP-сервера;
- состояние расчета хранится в базе, поэтому его можно восстановить после перезапуска сервиса.

## Компоненты

### Пользовательский Интерфейс

Папка:

```text
frontend/
```

Назначение:

- отображение карты пробоотбора;
- просмотр площадок и проб;
- добавление, изменение и удаление данных;
- фильтрация концентраций по году и металлу;
- запуск расчета;
- просмотр статуса и результата расчета;
- экспорт отчетов.

Интерфейс не выполняет расчетную логику и не работает с базой напрямую. Все операции выполняются через REST API.

Frontend отвечает только за пользовательский сценарий: показать карту, заполнить форму, отобразить таблицу, подсветить статус расчета и сформировать отчет. Если пользователь добавляет пробу или запускает расчет, интерфейс формирует JSON-запрос и передает его серверу.

### REST API

Папка:

```text
src/api/
```

Назначение:

- принимает HTTP-запросы;
- валидирует входные JSON-данные;
- возвращает JSON-ответы;
- управляет площадками, металлами и пробами;
- формирует статистические выборки;
- создает расчет и передает его в очередь обработки;
- отдает статус и результат расчета.

Основной файл маршрутов:

```text
src/api/routes.cpp
```

API является границей между внешним клиентом и внутренней логикой приложения. На этом уровне удобно выполнять проверку входных данных, приводить параметры к нужным типам, выбирать HTTP-коды ответа и не раскрывать клиенту внутреннюю структуру базы данных.

### Инфраструктурный Слой

Папка:

```text
src/infra/
```

Назначение:

- подключение к PostgreSQL;
- SQL-запросы;
- публикация сообщений в Kafka;
- чтение переменных окружения;
- конфигурация приложения.

Ключевые файлы:

- `postgres_repository.cpp`: работа с PostgreSQL;
- `kafka_producer.cpp`: отправка событий расчета;
- `config.cpp`: чтение настроек окружения.

Инфраструктурный слой содержит код, зависящий от конкретных технологий. Если в будущем поменяется способ хранения данных или брокер сообщений, основная расчетная логика останется изолированной: изменения будут сосредоточены вокруг repository и producer-классов.

### Доменный Слой

Папка:

```text
src/domain/
```

Назначение:

- расчет расстояний;
- парсинг числовых значений;
- расчет концентрации;
- генерация сетки точек загрязнения.

Доменный слой не зависит от HTTP, PostgreSQL или Kafka. Это позволяет тестировать расчетную логику отдельно.

Это центральная часть предметной области. Здесь находятся функции, которые описывают, как вычисляется расстояние, как интерпретируется числовое значение концентрации и как формируется расчетная сетка. Доменный слой можно запускать в unit-тестах без web-сервера, базы и Kafka.

### Worker

Папка:

```text
src/worker/
```

Назначение:

- получает событие о новом расчете;
- переводит расчет в статус `processing`;
- загружает исходные данные;
- выполняет расчет;
- сохраняет точки расчетной сетки;
- переводит расчет в статус `completed` или `failed`.

Worker нужен для того, чтобы тяжелый расчет не блокировал HTTP-запрос пользователя.

Worker работает как отдельный backend-процесс. Он не обслуживает браузерные запросы, а занимается только расчетными заданиями. Такой подход позволяет запускать несколько worker-процессов, если расчетов станет больше.

## Разделение Ответственности

Каждый слой выполняет свою задачу:

- React UI отвечает за отображение и действия пользователя;
- REST API отвечает за контракт приложения и безопасность входных данных;
- repository-слой отвечает за сохранение и чтение данных;
- Kafka producer отвечает за передачу задания на расчет;
- worker отвечает за выполнение задания;
- domain-слой отвечает за математическую часть;
- PostgreSQL отвечает за долговременное хранение состояния.

Такое разделение уменьшает связанность кода. Например, таблицу проб можно доработать во frontend без изменения формулы расчета, а формулу расчета можно проверить в тестах без запуска браузера.

## Жизненный Цикл Обычного Запроса

Обычная операция, например добавление пробы, выполняется синхронно:

```text
Пользователь заполняет форму
        |
        v
React отправляет POST-запрос
        |
        v
API проверяет JSON
        |
        v
Repository выполняет INSERT в PostgreSQL
        |
        v
API возвращает созданную запись
        |
        v
Интерфейс обновляет таблицу
```

Для пользователя такая операция выглядит мгновенной: он нажимает "Сохранить" и сразу видит обновленные данные.

## Жизненный Цикл Расчета

Расчет загрязнения выполняется асинхронно, потому что он может быть дольше обычного сохранения формы. API сначала создает job со статусом `queued`, затем отправляет событие в Kafka. Worker меняет статус на `processing`, выполняет расчет и сохраняет точки сетки. После успешного завершения статус становится `completed`.

Главное преимущество такого подхода в том, что HTTP-запрос запуска расчета быстро завершается, а пользователь может наблюдать статус отдельно.

## Поток Расчета

```text
Пользователь нажимает "Рассчитать"
        |
        v
POST /api/calculations/concentration
        |
        v
API создает расчет со статусом queued
        |
        v
API отправляет событие concentration.calculate.requested
        |
        v
Worker получает событие
        |
        v
Worker меняет статус на processing
        |
        v
Worker рассчитывает поле концентрации
        |
        v
Worker сохраняет точки в concentration_grid_points
        |
        v
Worker меняет статус на completed
        |
        v
Интерфейс загружает точки и показывает их на карте
```

## Конфигурация

Приложение настраивается через переменные окружения:

| Переменная | Назначение | Пример |
| --- | --- | --- |
| `TESTDIP_DATABASE_URL` | строка подключения к PostgreSQL | `postgresql://localhost:5432/testdip` |
| `TESTDIP_KAFKA_BROKERS` | адрес Kafka/Redpanda | `127.0.0.1:9092` |
| `TESTDIP_HTTP_PORT` | порт API | `8080` |
| `TESTDIP_FRONTEND_ROOT` | путь к собранному frontend | `/frontend` |

## Почему Расчет Выполняется Асинхронно

Расчет поля концентрации может занимать заметное время. Если выполнять его прямо внутри HTTP-запроса, пользовательский интерфейс будет ждать завершения операции, а серверный поток будет занят.

Асинхронная схема дает преимущества:

- пользователь быстро получает статус “в очереди”;
- расчет можно выполнять отдельным процессом;
- worker можно масштабировать;
- сбой расчета не останавливает API;
- история расчетов сохраняется в базе.

## Обработка Ошибок

В архитектуре предусмотрено хранение статуса расчета. Если задание не удалось отправить в очередь, оно получает статус `publish_failed`. Если ошибка возникла во время работы worker, расчет переходит в `failed`, а текст ошибки сохраняется в `error_message`.

Это важно для эксплуатации: пользователь видит не просто отсутствие результата, а конкретное состояние задания. Разработчик может проверить базу, логи API, логи worker и понять, на каком этапе возникла проблема.

## Масштабирование

Проект можно масштабировать постепенно:

- frontend можно отдавать через API или отдельный web-сервер;
- API можно запускать в нескольких экземплярах за балансировщиком;
- worker можно масштабировать отдельно от API;
- PostgreSQL остается центральным источником данных;
- Kafka/Redpanda выступает буфером между запуском расчета и его выполнением.

Для дипломного проекта достаточно одного API, одного worker, одной базы и одного брокера. Но выбранная схема уже показывает подход, который используется в реальных backend-системах.

## Тестируемость

Расчетная логика вынесена в `src/domain/`, поэтому она проверяется отдельно через `ctest`.

Тесты находятся в:

```text
tests/domain_tests.cpp
```

Проверяются:

- парсинг чисел;
- расчет расстояния;
- формула концентрации;
- генерация сетки концентраций.
